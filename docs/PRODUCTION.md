# Эксплуатация eTask в локальной сети

## Требования к серверу

- Linux x86_64, 4 CPU, 8 ГБ RAM и не менее 100 ГБ SSD;
- Docker Engine 26+ и Docker Compose v2;
- статический IP или внутренняя DNS-запись, например `etask.internal`;
- синхронизация времени NTP;
- отдельный каталог резервных копий на диске, который копируется на другой физический носитель.

Внешний доступ к портам PostgreSQL, API и frontend не публикуется. Для сотрудников открыт только `ETASK_HTTP_PORT` Nginx. Доступ к SSH разрешается только администраторам.

## Первый запуск

1. Создайте `.env` из `.env.example` и установите три независимых случайных секрета.
2. Укажите внутренний адрес в `FRONTEND_ORIGIN`.
3. Выполните `docker compose config`, затем `docker compose up -d --build`.
4. Проверьте `docker compose ps`: все сервисы должны быть `healthy`.
5. Войдите как системный администратор `admin` с `BOOTSTRAP_ADMIN_PASSWORD`, создайте персональные аккаунты директора и сотрудников. Bootstrap-администратор можно заменить персональным ИТ-аккаунтом после проверки его входа.

Системный администратор работает только в разделе «Администрирование»: создаёт и блокирует аккаунты, назначает роли и управления, сбрасывает пароли и просматривает технический журнал. Доступ к задачам, проектам, доскам, комментариям и рабочей аналитике для этой роли закрыт сервером. Директор управляет рабочими процессами, но не аккаунтами и паролями.

## TLS

Для HTTPS установите сертификат внутреннего удостоверяющего центра на reverse proxy, измените адрес на `https://...` и задайте `COOKIE_SECURE=true`. Закрытую систему также следует использовать только по HTTPS, если инфраструктура организации позволяет распространить корневой сертификат.

## Журналы

API пишет структурированные JSON-события в stdout без JWT, cookie и паролей. Docker хранит не более 10 файлов по 20 МБ на контейнер. Просмотр: `docker compose logs -f api gateway postgres`. Перед интеграцией с SIEM настройте сбор stdout Docker и сохранение поля `X-Request-Id`.

## Резервное копирование

Сервис `backup` сразу создаёт копию PostgreSQL и вложений, затем повторяет операцию каждые `BACKUP_INTERVAL_SECONDS`. Файлы сохраняются в `BACKUP_PATH`, сопровождаются SHA-256 и удаляются после `BACKUP_RETENTION_DAYS`.

Не реже раза в квартал проверяйте восстановление на отдельном сервере. Для восстановления остановите `api` и выполните:

```bash
docker compose stop api
docker compose --profile maintenance run --rm -e CONFIRM_RESTORE=YES restore /scripts/restore.sh /backups/etask-db-TIMESTAMP.dump
docker compose --profile maintenance run --rm -e CONFIRM_RESTORE=YES restore /scripts/restore-uploads.sh /backups/etask-uploads-TIMESTAMP.tar.gz
docker compose start api
```

Обе команды проверяют SHA-256 до изменения данных. Храните минимум одну копию вне основного сервера.

Операции копирования и восстановления выполняет системный администратор через защищённую SSH-сессию на сервере. Восстановление намеренно не запускается из браузера: оно останавливает API и заменяет рабочую базу, поэтому требует доступа к серверу, контрольной фразы и проверки контрольной суммы.

## Обновление

1. Сделайте резервную копию и проверьте SHA-256.
2. Примените новые SQL-миграции по порядку.
3. Соберите образ с фиксированным `ETASK_VERSION`, а не `latest`.
4. Выполните `docker compose up -d` и проверьте healthcheck и журналы.
5. При ошибке верните предыдущий тег образа и восстановите базу только при несовместимой миграции.
